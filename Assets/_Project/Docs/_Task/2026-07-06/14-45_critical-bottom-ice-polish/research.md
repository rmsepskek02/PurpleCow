# Research — Critical Text, Bottom Attack, and Ice Column Polish

이번 작업은 `TODO.md`의 1번(치명타 텍스트), 6번(바닥 도달 몬스터 연출), 5번(아이스볼 후방 전파)을 함께 구현하기 위한 사전 조사다. 세 기능의 현재 실행 흐름과 풀링·피해 처리 관계를 확인했으며, 구현 전에 사용자가 결정해야 하는 시각 수치와 판정 기준을 분리해 정리한다.

## 현재 상태

### 1. 치명타 데미지 텍스트

- `DamageTextFx`의 `_criticalColor` 기본값은 `Color.yellow`다.
- 실제 프리팹 `Assets/_Project/Prefabs/UI/DamageTextFx.prefab`에도 직렬화된 색상값이 있으므로, 스크립트 기본값만 바꾸면 기존 프리팹에는 반영되지 않는다.
- 치명타 여부에 따른 굵은 글씨와 1.5배 크기 연출은 이미 구현되어 있어 색상만 변경하면 된다.

### 5. 아이스볼 후방 몬스터 동시 정지

- `IceBallSkill.OnBallHit()`은 확률 판정 성공 시 피격 대상 하나에 Freeze, Slow, 추가 피해를 적용한다.
- `WaveManager`는 활성 몬스터 목록을 소유하며 같은 행을 찾는 `GetMonstersInRow()`만 제공한다.
- 몬스터는 스폰 이후 그리드 칸에 고정되지 않고 매 프레임 아래로 이동한다.
- 그리드 한 칸 간격은 `_gridCellSize = 1.0`이다.
- 몬스터 크기는 1×1, 2×1, 1×2 세 종류이므로 중심점 x좌표만 비교할 경우 가로 2칸 몬스터가 실제로 차지하는 열을 정확히 반영하지 못할 수 있다.
- 전파 목적은 공격 범위 확장이 아니라 정지된 몬스터와 뒤에서 내려오는 몬스터의 시각적 겹침 방지다. 따라서 최초 대상과 같은 수평 점유 범위에 있고, 대상보다 위쪽에 있는 활성 몬스터만 전파 대상으로 삼는 것이 목적에 가장 부합한다.
- 추가 피해는 최초 피격 대상에게만 적용하고, 후방 몬스터에는 Freeze와 Slow만 전파해야 아이스볼의 공격력이 열 전체에 중복 적용되는 것을 피할 수 있다.

### 6. 바닥 도달 몬스터 공격 연출

- `WaveManager.CheckGameOver()`는 몬스터가 `_bottomBoundaryY`에 도달하면 다음 순서로 처리한다.
  1. 활성 목록에서 제거
  2. 오브젝트 풀에 반환하여 비활성화
  3. `OnMonsterReachedBottom` 이벤트 발행
- `CharacterManager`는 위 이벤트를 받아 즉시 체력을 감소시키고 몬스터 보상 XP를 지급한다.
- 현재 이벤트는 몬스터가 풀에 반환된 뒤 발행된다. 새 연출에서는 비활성화된 몬스터를 움직일 수 없으므로 처리 순서를 반드시 바꿔야 한다.
- 캐릭터 비주얼 오브젝트는 `LaunchPoint/Character` 구조이며, `CharacterManager` 오브젝트와는 별개다. 돌진 목표로 정확한 캐릭터 위치를 사용하려면 `WaveManager` 또는 몬스터 연출 컴포넌트에 캐릭터 Transform 참조가 필요하다.
- 씬의 네이티브 기준값은 하단 벽 `-7.5`, 발사 지점 `-6.7`, 캐릭터 로컬 Y `-0.4`이며 `WallFitter`가 화면 비율에 맞춰 월드 좌표를 보정한다. 따라서 고정 좌표보다 실제 캐릭터 Transform을 목표로 사용해야 해상도 대응이 유지된다.
- 연출 중 몬스터를 활성 목록에서 즉시 제거하면, 실제 충돌 전에 웨이브 클리어와 다음 웨이브 시작이 발생할 수 있다.
- 반대로 활성 목록에만 남겨 두고 별도 상태가 없으면 `CheckGameOver()`가 매 프레임 연출을 중복 시작한다.
- 연출 중 볼에 다시 맞아 사망하면 풀 반환과 Tween 완료 콜백이 충돌할 수 있다. 바닥 공격을 시작한 몬스터는 이동과 피격을 중단하고, 충돌 완료 시에만 피해 이벤트·풀 반환·웨이브 상태 갱신을 수행하는 별도 상태가 필요하다.
- `ObjectPool.Return()`은 `OnDespawn()` 호출 후 GameObject를 비활성화한다. 진행 중 Tween은 `OnDespawn()`에서 확실히 종료하고, 재사용 시 위치·스케일·회전·Collider 상태를 복구해야 한다.

## 문제점 / 구현 대상 파악

### 치명타 색상

스크립트 기본값과 프리팹 직렬화 값을 함께 변경해야 한다. 정확한 붉은 계열 색상은 아직 확정되지 않았다.

### 아이스볼 열 판정

단순 `Mathf.Abs(x 차이) < 0.1f`는 1×1 몬스터에는 충분하지만 2×1 몬스터의 점유 폭을 반영하지 못한다. 다음 두 방식 중 하나를 확정해야 한다.

1. **실제 수평 점유 범위 중첩 방식(권장)**  
   두 몬스터 Collider의 x축 범위가 겹치면 같은 열로 판단한다. 2×1 몬스터도 실제 차지한 두 열을 자연스럽게 포함한다.
2. **중심 x좌표 허용 오차 방식**  
   구현은 단순하지만 2×1 몬스터가 열 판정에서 누락되거나 예상보다 넓게 묶일 수 있다.

후방은 현재 이동 방향이 아래쪽이므로 `candidate.y > target.y`인 몬스터로 정의할 수 있다. 최초 대상은 기존 방식대로 효과 지속시간을 갱신하고, 후방 대상도 같은 Freeze/Slow 값을 적용한다.

### 바닥 공격 상태와 처리 순서

안전한 흐름은 다음과 같다.

1. 바닥 도달 감지
2. 몬스터를 `BottomAttacking` 상태로 전환하고 일반 이동·피격·Collider 상호작용 중단
3. 제자리 진동
4. 실제 캐릭터 Transform 위치로 돌진
5. 도착 시 `OnMonsterReachedBottom` 동기 이벤트 발행
6. `CharacterManager`가 그 순간 HP 감소 및 기존과 동일한 XP 처리
7. 이벤트 처리 후 활성 목록 제거 및 풀 반환
8. 몬스터 수·웨이브 클리어 상태 갱신

이 순서라면 피해가 돌진 시작이 아닌 충돌 순간 적용되고, 이벤트 처리 중에도 몬스터 데이터가 유효하다.

## 후속 진단 — 아이스볼 후방 겹침 재발

최초 구현 후 사용자 플레이 테스트에서 아이스볼에 의해 정지한 몬스터와 뒤따르는 몬스터가 다시 겹치는 현상이 확인되었다.

### 재현 조건

- 씬의 그리드 간격은 `0.85`다.
- 네 종류 몬스터의 이동속도는 모두 초당 `0.2`다.
- 다음 스폰 검사 주기는 `0.85 / 0.2 = 4.25초`다.
- 아이스볼 빙결 지속시간은 레벨별 `5 / 6 / 7초`다.
- 현재 `IceBallSkill`은 아이스볼 적중 순간 `GetMonstersBehindInColumn()`으로 조회된 몬스터에만 Freeze/Slow를 한 번 적용한다.

적중 당시 존재하던 후방 몬스터는 함께 멈추므로 간격이 유지된다. 하지만 적중 직후 같은 열에 새 몬스터가 배치되면 그 몬스터는 최초 조회 목록에 없기 때문에 계속 이동한다.

- Lv.1에서 즉시 새로 배치된 몬스터는 빙결 종료 전 최대 `1.0`만큼 이동할 수 있다.
- Lv.2에서는 최대 `1.2`, Lv.3에서는 최대 `1.4`만큼 이동할 수 있다.
- 모두 기본 간격 `0.85`보다 크므로 정지 몬스터와 실제로 겹칠 수 있다.

### 가설 판별

1. **적중 이후 등장한 몬스터가 정지 대상에서 누락된다 — 확인됨**
   - 현재 조회와 상태이상 적용은 적중 순간 한 번만 실행된다.
   - 빙결 지속시간이 스폰 검사 주기보다 길어 새 몬스터가 빙결 중 합류할 수 있다.
2. **몬스터별 이동속도 차이로 추월한다 — 기각**
   - 네 종류 모두 `_moveSpeed = 0.2`다.
3. **Collider 가로 범위 판정이 같은 열을 놓친다 — 주원인 아님**
   - 최초 목록은 실제 Collider 가로 중첩으로 판정한다.
   - 사용자 증상은 최초 판정 이후의 시간 흐름에서도 발생할 수 있다.
4. **후방 몬스터의 빙결 시간이 직접 대상보다 짧다 — 기각**
   - 같은 프레임에 같은 `LevelData.Value2`를 적용한다.

### 권장 수정

적중 순간의 대상 목록을 늘리는 방식만으로는 이후 스폰을 막을 수 없다. 몬스터가 아래로 이동하기 직전에, 자신보다 아래쪽 같은 열에 아직 빙결 중인 몬스터가 있는지 동적으로 확인하는 방식이 필요하다.

1. `WaveManager`에 같은 Collider 가로 범위의 아래쪽 빙결 몬스터 존재 여부를 조회하는 메서드를 추가한다.
2. `MonsterBase.Update()`에서 일반 이동 직전에 해당 조건을 조회한다.
3. 아래쪽에 빙결 몬스터가 있으면 그 프레임에는 이동하지 않는다.
4. 빙결이 끝나거나 해당 몬스터가 처치되면 자동으로 이동을 재개한다.

이 방식은 아이스볼 적중 이후 새로 배치된 몬스터도 즉시 멈추며, 1×1·1×2·2×1 Collider 점유 폭을 그대로 활용한다. 물리 Collider 충돌에 맡기지 않으므로 Transform 직접 이동 구조에서도 동작하고, 별도 장기 상태나 반복 코루틴이 필요 없다.

## 후속 요구 — 바닥 공격 연출 Inspector 조정

현재 바닥 공격 수치는 `WaveManager`의 상수로 고정되어 있다.

- 진동 시간 `0.35`
- 진동 강도 `0.12`
- 돌진 시간 `0.25`

사용자는 진동 관련 수치를 Inspector에서 직접 조정할 수 있도록 요청했다. 시각 연출을 한 위치에서 조정할 수 있도록 세 값을 `WaveManager`의 `[SerializeField] private` 필드로 전환하는 안을 권장한다. 기존 씬에는 현재 확정값 `0.35 / 0.12 / 0.25`를 그대로 직렬화해 화면 결과가 갑자기 바뀌지 않게 한다.

## 후속 결론

아이스볼 겹침의 원인은 최초 적중 시점의 일회성 후방 조회로 확정되었다. 동적 이동 차단 방식으로 보완해야 이후 스폰까지 안전하게 처리할 수 있다. Inspector 노출 범위는 진동 시간·강도만 노출할지, 같은 연출의 돌진 시간까지 함께 노출할지 사용자 확인 후 `plan.md`를 보완한다.

## 추가 후속 진단 — 스폰 시점 Collider 겹침

사용자 플레이 테스트에서 아이스볼로 정지한 몬스터를 따라오는 경우뿐 아니라, 새 몬스터가 스폰되는 순간부터 기존 몬스터와 겹치는 현상도 확인되었다.

### 현재 점유 검사

`WaveManager.IsCellFree()`는 스폰 셀 중심과 활성 몬스터 Transform 중심의 거리를 비교한다.

- 씬 그리드 셀 크기: `0.85`
- 검사 반경: `0.85 × 0.55 = 0.4675`
- 1×1 몬스터 Collider 원본 크기: `0.96`
- 프리팹 루트 배율: `0.85 / 0.96`
- 실제 월드 Collider 한 칸 크기: 약 `0.85`

기존 몬스터 중심이 스폰 셀 중심에서 아래로 `0.5` 이동했다고 가정하면 현재 검사는 `0.5 >= 0.4675`이므로 해당 셀을 비어 있다고 반환한다. 그러나 기존 몬스터와 새 몬스터 Collider의 세로 반지름 합은 `0.425 + 0.425 = 0.85`이므로 실제로는 `0.85 - 0.5 = 0.35`만큼 겹친다. 이는 사용자에게 보이는 “겹친 상태로 스폰”을 코드 수치만으로 재현한다.

빙결이 없을 때는 스폰 검사 주기 동안 앞 몬스터가 한 칸만큼 이동해 우연히 간격이 맞는다. 하지만 아이스볼이 몬스터를 셀 중심에서 `0.4675~0.85` 사이만 내려간 위치에 정지시키면 중심점 검사는 빈 칸으로 오판하면서 Collider는 여전히 스폰 셀을 차지한다.

### 가설 판별

1. **중심점 검사 반경이 실제 Collider 점유 범위보다 작다 — 확인됨**
   - `0.4675`보다 멀지만 Collider 분리 거리 `0.85`보다 가까운 구간이 존재한다.
2. **2칸 몬스터의 실제 전체 점유 범위를 직접 비교하지 않는다 — 확인됨**
   - 현재 코드는 중심점과 셀별 근사 반경만 사용한다.
   - 2×1·1×2 몬스터는 실제 Collider 폭·높이를 직접 비교하는 편이 정확하다.
3. **같은 스폰 틱에서 `topRowFree` 갱신이 누락된다 — 기각**
   - 배치 직후 BlockSize별 점유 셀을 `false`로 변경한다.
4. **1·11웨이브 전체 그리드 배치 배열이 중복 칸을 허용한다 — 기각**
   - `free[,]` 배열을 BlockSize별로 즉시 갱신한다.

### 권장 수정

`IsCellFree()`의 중심점 거리 근사를 실제 Collider Bounds 교집합 판정으로 교체한다.

1. 검사 대상 셀 중심과 `_gridCellSize`로 월드 셀 Bounds를 만든다.
2. 각 활성 몬스터의 실제 `BoxCollider2D.bounds`를 가져온다.
3. x축과 y축 모두에서 양수 폭으로 겹치면 점유 중으로 판정한다.
4. 경계가 정확히 닿기만 하는 경우에는 겹침으로 처리하지 않아 한 칸 간격의 정상 배치는 허용한다.
5. 기존 `TryDispenseRoster()`의 1×1은 한 셀, 2×1은 가로 두 셀, 1×2는 세로 두 셀 검사 구조를 유지한다.

이를 위해 `MonsterBase`의 기존 수평 Bounds 조회를 전체 Collider Bounds 조회로 확장하거나 공통 메서드를 추가한다. 동적 빙결 열 차단과 스폰 점유 검사가 동일한 실제 Collider 정보를 사용하게 하면 두 겹침 원인을 같은 기준으로 해결할 수 있다.

## 추가 후속 결론

아이스볼 관련 겹침에는 두 독립 원인이 함께 존재한다.

1. 적중 이후 새로 합류한 후방 몬스터가 일회성 Freeze 목록에 포함되지 않음
2. 스폰 셀 점유 검사가 실제 Collider가 아니라 작은 중심점 반경을 사용해 부분 겹침을 빈 칸으로 오판함

따라서 동적 후방 이동 차단과 Collider Bounds 기반 스폰 점유 검사를 함께 적용해야 한다.
